CMS User Journey Map

콘텐츠 업무 흐름 3가지 시나리오 + 권한 관리 설계 기초

프로젝트: CMS 고도화 (NDCM) 프로토타입: v8.5 작성일: 2026-05-19 작성자: Shawn (플랫폼기획팀)
Journey 1 신규 콘텐츠 등록 — 처음부터 배포까지
콘텐츠가 존재하지 않는 상태에서 양식 정의, 커리큘럼 설계, 콘텐츠 제작, 검수, 과목코드 연결, 배포까지 전체 워크플로우를 추적합니다.
CMS의 핵심 원칙인 "콘텐츠 독립성" 아키텍처에 따라 콘텐츠는 과목코드와 독립적으로 제작되며, 이후 연결 단계에서 서비스 시스템에 매핑됩니다.

Phase 0 — 준비 양식 관리 / 커리큘럼 관리
1
기획 관리자
입력 양식 템플릿 생성
콘텐츠 입력에 사용할 필드 구성을 정의한다. 문항번호, 정답, 배점, 해설 등 필드를 설계하고 양식 이름과 코드를 부여한다.
화면: 입력 양식 관리 (form-manage)
2
기획 관리자
커리큘럼 생성 (빌더)
기본정보(과목, 학년, 교육과정) 입력 후 단계 구조(커리큘럼명→과정→세트→콘텐츠)를 설계한다. 외부 수신 데이터에서 구조를 가져오거나 균등 배분으로 생성할 수 있다.
화면: 커리큘럼 관리 > 빌더 (curriculum)
3
기획 관리자
양식 연결 + 분할 기준 설정
커리큘럼에 사용할 양식을 연결하고, 작업 분할 기준 단계(예: 세트 레벨)를 지정한다. 동일 커리큘럼에서 대상 시스템별로 다른 양식을 연결할 수 있다.
화면: 커리큘럼 관리 > 빌더 섹션2 (curriculum)
▼
기획 관리자
커리큘럼 활성화 (draft → active)
유효성 검증: 단계 2개 이상, 양식 1개 이상, 분할 기준 지정 여부를 확인한다.
통과 → 자동으로 작업 설정 화면 이동
실패 → 빌더에서 누락 항목 보완 필요
▼
Phase 1 — 배정 (assign) 작업 설정 (work-config)
4
기획 관리자
담당자 배정 + 작업 범위 지정
커리큘럼 카드를 선택하고, 양식별로 콘텐츠 담당자를 배정한다. 분할 기준 단계(세트)에서 작업 범위(예: 1~40번)를 지정하면 WORK_ITEMS_V7 레코드가 생성된다.
화면: 작업 설정 > 작업 배정 모달 (work-config)
▼
Phase 2 — 등록 (register) 작업 진행 보드 / 콘텐츠 등록
5
콘텐츠 담당자
배정된 작업 확인
작업 진행 보드(칸반)의 '등록' 컬럼에서 자신에게 배정된 작업 카드를 확인한다. 카드에는 커리큘럼명, 양식, 작업 범위, 진행률이 표시된다.
화면: 작업 진행 보드 (workflow)
6
콘텐츠 담당자
콘텐츠 편집기에서 입력
콘텐츠 등록 화면에서 해당 양식의 편집기를 열고, 문항별로 콘텐츠를 입력한다. 파일 업로드(이미지, 오디오 등)와 저장소 관리를 수행한다. 작업 진행률이 실시간 반영된다.
화면: 콘텐츠 등록 > 편집기 (content-register)
7
콘텐츠 담당자
등록 완료 → 검수 요청
모든 세트 입력 완료 시 '검수 요청' 버튼으로 작업을 검수 단계로 이동시킨다. WORK_ITEMS_V7.phase가 register → review로 전환된다.
화면: 콘텐츠 등록 (content-register)
▼
Phase 3 — 검수 (review) 검수 관리
8
검수자
검수 뷰어에서 콘텐츠 확인
검수 관리 화면에서 검수 대기 작업을 선택하고, 검수 뷰어에서 콘텐츠를 미리보기한다. 체크리스트를 확인하고 코멘트를 남긴다.
화면: 검수 관리 > 검수 뷰어 (review)
검수자
검수 판정
콘텐츠 품질, 정확성, 완성도를 평가한다.
승인 → phase: review → deploy 전환
반려 → phase: review → register 복귀 (재작업)
▼
Phase 4 — 과목코드 연결 과목코드 연결
9
기획 관리자
양식별 과목코드 연결
검수 완료된 커리큘럼의 양식별로 서비스 시스템(뉴드림스, 눈높이 등)의 과목코드를 연결한다. 전체(full) 또는 부분(partial, 범위 지정) 연결이 가능하다.
화면: 과목코드 연결 (service-bind)
▼
Phase 5 — 배포 (deploy) 배포 관리
10
기획 관리자
배포 스케줄 생성
과목코드 연결이 완료된 작업을 배포 관리 화면에서 배포 스케줄에 등록한다. 스테이징(STG) 배포 후 운영(PRD) 배포를 진행한다.
화면: 배포 관리 (deploy)
11
기획 관리자
배포 완료 → 서비스 전환
PRD 배포 완료 시 해당 콘텐츠는 서비스 콘텐츠 관리에서 'live' 상태로 전환된다. 배포 이력(버전, 패치노트)이 기록된다.
화면: 배포 관리 → 서비스 콘텐츠 관리 (deploy → service-content)

Journey 1 핵심 포인트

  • 콘텐츠는 과목코드와 완전히 독립적으로 제작된다 (콘텐츠 독립성 아키텍처)
  • 양식(Form)이 시스템 전달 단위 — 동일 콘텐츠라도 대상 시스템별로 별도 양식으로 존재
  • 커리큘럼 활성화(draft→active) 시 유효성 게이트가 작동한다
  • WORK_ITEMS_V7이 전체 워크플로우의 단일 진실 소스(Single Source of Truth)
  • 검수 반려 시 등록 단계로 돌아가며, 반복 루프가 가능하다
  • 과목코드 연결은 배포 전 필수이며, 연결 완료된 콘텐츠만 배포할 수 있다
Journey 2 콘텐츠 수정 — 사용 전 (미배포 상태)
아직 서비스에 배포되지 않은 콘텐츠를 수정하는 워크플로우입니다. 과목코드 미연결 또는 배포 전 단계에서 발생하며,
기존 작업 항목을 찾아 수정 후 재검수→배포 프로세스를 거칩니다. 버전 관리 없이 직접 수정(덮어쓰기)합니다.

진입 조건

콘텐츠 상태등록 완료 또는 검수 반려 (미배포)
과목코드미연결 또는 연결 완료 (배포 전)
서비스 상태미운영 (live 아님)
수정 방식직접 편집 (덮어쓰기, 버전 관리 없음)

주의사항

데이터 영향원본 데이터가 직접 수정됨
검수 필요수정 후 재검수 필수
워크플로우phase가 register로 복귀됨
권한배정된 담당자 또는 관리자만 수정 가능

콘텐츠 담당자
수정 대상 작업 확인
작업 진행 보드에서 해당 작업 카드를 찾거나, 콘텐츠 등록 화면에서 커리큘럼 카드를 통해 수정할 작업을 선택한다.
화면: 작업 진행 보드 (workflow) 또는 콘텐츠 등록 (content-register)
▼
수정 작업 콘텐츠 등록 > 편집기
2
콘텐츠 담당자
편집기에서 콘텐츠 수정
기존 콘텐츠를 직접 수정한다. 필드값 변경, 파일 교체, 오류 수정 등을 수행한다. 수정 시 phase가 자동으로 register로 전환된다.
화면: 콘텐츠 등록 > 편집기 (content-register)

콘텐츠 담당자
수정 완료 → 재검수 요청
수정이 끝나면 '검수 요청' 버튼으로 다시 검수 단계로 이동시킨다.
화면: 콘텐츠 등록 (content-register)
▼
재검수 검수 관리
4
검수자
변경 사항 검수
수정된 콘텐츠를 검수 뷰어에서 확인한다. 변경 사항이 올바른지 검토하고 승인/반려를 결정한다.
화면: 검수 관리 > 검수 뷰어 (review)
검수 판정
승인 시 배포 단계로, 반려 시 등록 단계로 복귀
▼
연결 확인 과목코드 연결
5
기획 관리자
과목코드 연결 확인/신규 연결
이미 연결된 경우 변경 사항 없이 통과. 미연결 상태라면 이 시점에서 양식별 과목코드를 연결한다.
화면: 과목코드 연결 (service-bind)

배포 배포 관리

기획 관리자
STG → PRD 배포
스테이징 배포 후 운영 배포를 진행한다.
화면: 배포 관리 (deploy)
7
기획 관리자
배포 완료 → 서비스 전환
배포 완료 시 서비스 콘텐츠 관리에서 live 상태로 전환된다.
화면: 서비스 콘텐츠 관리 (service-content)
Journey 3 콘텐츠 수정 — 사용 중 (서비스 운영 상태)
이미 서비스에 배포되어 운영 중인 콘텐츠를 수정하는 워크플로우입니다. 운영 중인 콘텐츠는 직접 수정할 수 없으며,
개정(revision) 생성을 통해 새 버전을 만들고, 수정→검수→재배포 과정을 거칩니다. 문제 발생 시 롤백이 가능합니다.

진입 조건

콘텐츠 상태배포 완료, 서비스 운영 중 (live)
과목코드연결 완료
서비스 상태live (운영 중)
수정 방식개정(revision) 생성 → 새 버전으로 수정

핵심 차이점 (vs 사용 전)

원본 보호운영 중 원본은 수정 불가 (읽기 전용)
버전 관리v1.0 → v1.1 개정본 자동 생성
롤백배포 후 문제 발생 시 이전 버전 복원 가능
영향 범위연결된 모든 서비스 시스템에 영향
진입 서비스 콘텐츠 관리
1
기획 관리자
서비스 콘텐츠 현황 확인
서비스 콘텐츠 관리 화면에서 운영 중인 커리큘럼과 양식 목록을 확인한다. 현재 버전, 배포일, 연결 시스템 정보를 조회한다.
화면: 서비스 콘텐츠 관리 (service-content)
2
기획 관리자
개정(revision) 생성
수정이 필요한 양식을 선택하고 '개정 생성' 버튼을 클릭한다. 현재 운영 버전을 복사한 새 개정본(v1.0 → v1.1)이 생성되며, 상태가 'revision'으로 전환된다.
화면: 서비스 콘텐츠 관리 (service-content)
▼
개정 작업 콘텐츠 등록 > 편집기
3
콘텐츠 담당자
개정본 편집
콘텐츠 등록 편집기에서 개정본을 열고 필요한 수정을 진행한다. 편집기 상단에 현재 버전(v1.1)과 이전 버전(v1.0) 정보가 표시된다. 원본(v1.0)은 운영 상태 유지.
화면: 콘텐츠 등록 > 편집기 (content-register)
4
콘텐츠 담당자
수정 완료 → 검수 요청
개정본 수정 완료 후 검수 단계로 이동시킨다.
화면: 콘텐츠 등록 (content-register)
▼
검수 검수 관리
5
검수자
개정본 검수
검수 뷰어에서 개정본의 변경 사항을 확인한다. 이전 버전과의 차이를 검토하고 승인/반려를 결정한다.
화면: 검수 관리 > 검수 뷰어 (review)
검수 판정
승인 시 재배포, 반려 시 개정본 재수정
▼
재배포 배포 관리
6
기획 관리자
개정본 배포 (기존 버전 대체)
배포 관리에서 개정본을 배포한다. STG 검증 후 PRD 배포 시 기존 운영 버전(v1.0)이 개정본(v1.1)으로 대체된다. 패치노트를 기록한다.
화면: 배포 관리 (deploy)
7
기획 관리자
서비스 콘텐츠 상태 갱신
배포 완료 시 서비스 콘텐츠 관리에서 버전이 v1.1로 업데이트되고, 상태가 다시 'live'로 전환된다.
화면: 서비스 콘텐츠 관리 (service-content)
▼
롤백 (예외) 배포 관리
8
기획 관리자
롤백 (문제 발생 시)
배포 후 문제가 발견되면 배포 이력에서 이전 버전(v1.0)을 선택하여 롤백한다. 롤백 시 즉시 이전 버전으로 복원되며, 개정본은 별도 보관된다.
화면: 배포 관리 > 배포 이력 (deploy)
9
기획 관리자
롤백 후 개정본 재수정
롤백된 후 개정본을 다시 수정하여 재배포 프로세스(3~7단계)를 반복한다.
화면: 콘텐츠 등록 → 검수 → 배포 (전체 루프)

Journey 3 핵심 포인트

  • 운영 중 콘텐츠는 직접 수정 불가 — 개정(revision) 생성을 통해서만 수정
  • 개정본 작업 중에도 기존 운영 버전(v1.0)은 그대로 서비스 유지
  • 배포 이력에 모든 버전이 기록되며, 언제든 이전 버전으로 롤백 가능
  • 과목코드 재연결은 불필요 — 기존 연결 설정이 그대로 유지됨
  • 연결된 모든 서비스 시스템에 동시 영향 — 배포 전 충분한 검증 필요
역할 정의 CMS 사용자 역할 (Role) 설계
3가지 Journey에서 도출된 액터를 기반으로 CMS 시스템의 역할을 정의합니다.
각 역할은 접근 가능한 화면, 수행 가능한 액션, 데이터 범위가 구분됩니다.
SA
시스템 관리자
System Administrator
시스템 전체 설정 및 사용자 관리를 담당한다. 공지사항 관리, 계정 생성/비활성화, 역할 배정, 시스템 설정 변경 등 관리 기능에 접근한다. 일반 콘텐츠 업무는 수행하지 않는다.
시스템 관리 CRUD 사용자 관리 공지사항 관리 전체 읽기
PM
기획 관리자
Planning Manager
콘텐츠 워크플로우의 전체 관리자. 양식 정의, 커리큘럼 설계, 작업 배정, 과목코드 연결, 배포를 담당한다. 모든 작업 항목의 상태를 조회하고 전환할 수 있으며, 검수를 제외한 전 단계에 걸쳐 권한을 가진다.
양식 CRUD 커리큘럼 CRUD 작업 배정 과목코드 연결 배포 관리 서비스 콘텐츠 관리
CW
콘텐츠 담당자
Content Worker
실제 콘텐츠를 입력하고 수정하는 작업자. 자신에게 배정된 작업만 조회하고 편집할 수 있다. 작업 진행 보드에서 자신의 작업을 확인하고, 콘텐츠 등록 편집기에서 문항별 콘텐츠를 입력한다. 작업 완료 후 검수를 요청한다.
내 작업 조회 콘텐츠 입력/수정 파일 업로드 검수 요청
QA
검수자
Quality Reviewer
콘텐츠 품질을 검증하는 역할. 검수 대기 작업을 확인하고 검수 뷰어에서 콘텐츠를 리뷰한다. 승인/반려 결정 권한을 가지며, 반려 시 코멘트를 남겨 수정 방향을 안내한다. 콘텐츠를 직접 수정할 수는 없다.
검수 대기 조회 검수 뷰어 열람 승인/반려 결정 코멘트 작성

권한 설계 핵심 원칙

  • 최소 권한 원칙 (Least Privilege) — 각 역할에 업무 수행에 필요한 최소한의 권한만 부여
  • 직무 분리 (Separation of Duties) — 콘텐츠 작성자(CW)와 검수자(QA)를 분리하여 품질 게이트 보장
  • 데이터 격리 (Data Isolation) — CW는 본인 배정 데이터만 접근. 타인의 작업 데이터 열람 불가
  • 관리 권한 집중 (Centralized Administration) — 구조 설계(양식/커리큘럼), 배포, 과목코드 연결은 PM에 집중
  • 감사 추적 (Audit Trail) — 모든 상태 전환(phase 변경, 검수 판정, 배포)에 액터와 시간 기록
