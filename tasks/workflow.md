# 워크플로우 설계

## 1. 기본은 계획 모드
- 사소하지 않은 모든 작업은 계획 모드로 시작 (3단계 이상이거나 구조적 결정이 필요한 경우)
- 뭔가 잘못되면 즉시 멈추고 다시 계획 세우기 — 그냥 밀어붙이지 말 것
- 계획 모드는 개발할 때만이 아니라 검증 단계에서도 활용
- 모호함을 줄이려면 처음부터 상세 스펙을 작성할 것

## 2. 서브에이전트 활용
- 메인 컨텍스트 창을 깔끔하게 유지하려면 서브에이전트를 적극적으로 활용
- 조사, 탐색, 병렬 분석은 서브에이전트에 맡길 것
- 복잡한 문제일수록 서브에이전트를 더 많이 투입
- 서브에이전트 하나당 하나의 작업만 — 집중 실행을 위해

## 3. 자기개선 루프
- 사용자에게 수정을 받을 때마다: 해당 패턴을 `tasks/lessons.md`에 기록
- 같은 실수를 반복하지 않도록 스스로 규칙을 작성
- 실수율이 낮아질 때까지 이 교훈들을 반복해서 다듬을 것
- 세션 시작 시 해당 프로젝트의 교훈 목록을 먼저 검토

## 4. 완료 전 반드시 검증
- 작동한다는 걸 증명하기 전까지 절대 작업 완료로 표시하지 말 것
- 필요한 경우 main과 변경 사항 간의 동작 차이를 비교
- 스스로에게 물어볼 것: "시니어 엔지니어가 이걸 승인할까?"
- 테스트 실행, 로그 확인, 정확성 입증

## 5. 우아함을 추구할 것 (균형 있게)
- 사소하지 않은 변경이라면: 잠깐 멈추고 "더 우아한 방법이 있지 않을까?" 자문
- 수정이 어설프게 느껴진다면: "지금 내가 아는 모든 걸 바탕으로, 우아한 해결책을 구현해"
- 단순하고 명백한 수정엔 이 과정을 생략 — 오버엔지니어링 금지
- 결과물을 내놓기 전에 스스로 검토

## 6. 자율적인 버그 수정
- 버그 리포트를 받으면: 그냥 고칠 것. 하나하나 물어보지 말 것
- 로그, 에러, 실패한 테스트를 직접 찾아서 해결
- 사용자가 컨텍스트를 전환할 필요가 없도록
- 어떻게 하라는 말 없이도 실패한 CI 테스트를 직접 가서 고칠 것

# 작업 관리

1. **계획 먼저**: 체크 가능한 항목으로 `tasks/todo.md`에 계획 작성
2. **계획 검토**: 구현 시작 전에 확인
3. **진행 상황 추적**: 진행하면서 완료 항목 표시
4. **변경 사항 설명**: 각 단계마다 상위 수준 요약 제공
5. **결과 문서화**: `tasks/todo.md`에 검토 섹션 추가
6. **교훈 기록**: 수정 후 `tasks/lessons.md` 업데이트

# 핵심 원칙

- **단순함 우선**: 모든 변경은 최대한 단순하게. 건드리는 코드는 최소화.
- **게으름 금지**: 근본 원인을 찾을 것. 임시방편 없음. 시니어 개발자 기준으로.
- **최소 영향**: 꼭 필요한 부분만 수정. 새로운 버그를 만들지 말 것.

# 트리거 경계 (Harness 패턴 차용)

작업 유형별로 **작동해야 할 때(SHOULD)** 와 **작동하지 말아야 할 때(NOT)** 를 명확히 구분한다.

## 새 뷰 추가
- **SHOULD**: 사이드바 메뉴에 새 항목 필요, ALL_VIEWS 등록 필요, switchView 연결 필요
- **NOT**: 기존 뷰 안에 탭/섹션만 추가하는 경우 (뷰 내부 수정으로 처리)

## Python patch 스크립트 사용
- **SHOULD**: 10줄 이상의 함수 교체, 여러 위치에 걸친 패턴 치환, CSS 블록 전체 교체
- **NOT**: 단일 위치 수정, 텍스트 오타 교정, 속성값 변경 (Edit 도구로 충분)

## CLAUDE.md 동기화
- **SHOULD**: 사이드바 메뉴 변경, 데이터 구조 추가/변경, 새 뷰 추가, 아키텍처 변경
- **NOT**: UI 미세조정, 색상 변경, 버그 수정, 텍스트 교정

## 서브에이전트 위임
- **SHOULD**: 코드베이스 전체 패턴 검색, 여러 파일 교차 분석, 병렬 가능한 독립 조사
- **NOT**: 단일 함수 수정, 이미 위치를 아는 코드 편집, 순차 의존성이 있는 작업

# 테스트 시나리오 (작업 완료 전 검증)

모든 구현은 아래 3가지 시나리오를 머릿속에서 검증한 후 완료 표시한다.

## ① 정상 시나리오 (Happy Path)
- 새 기능이 의도한 대로 동작하는가?
- 샘플 데이터로 렌더링이 정상인가?
- 클릭/인터랙션이 올바르게 반응하는가?

## ② 기존 상태 시나리오 (Existing State)
- 기존 뷰/기능에 영향이 없는가? (사이드바 다른 메뉴 클릭 테스트)
- 기존 데이터 구조와의 호환성이 유지되는가?
- switchView 전환이 정상인가?

## ③ 에러/엣지 시나리오 (Error/Edge)
- 빈 데이터(배열 0건)일 때 빈 상태 UI가 표시되는가?
- 유니코드 문자열이 깨지지 않는가?
- JS syntax check가 통과하는가?

# 에러 처리 전략

## Python patch 스크립트 실패 시
1. **재시도**: 마커 문자열 확인 후 정확한 매칭으로 재작성
2. **스킵**: 해당 부분만 건너뛰고 나머지 적용, 실패 부분은 Edit 도구로 수동 처리
3. **폴백**: patch 포기하고 전체 함수를 새로 Write

## Edit 도구 매칭 실패 시
1. **재시도**: 더 넓은 컨텍스트(전후 3줄)로 old_string 확대
2. **스킵**: 해당 편집을 건너뛰고, Read로 현재 상태 재확인
3. **폴백**: Python patch 스크립트로 전환

## JS syntax check 실패 시
1. **진단**: 에러 메시지에서 라인 번호 추출 → 해당 위치 Read
2. **수정**: 구문 오류 직접 수정 (보통 따옴표/괄호 불일치)
3. **재검증**: 수정 후 반드시 다시 syntax check 실행

# 의존성 순서 (DAG)

작업 유형별 실행 순서를 정의한다. `→`는 선행 의존, `||`는 병렬 가능.

## 새 뷰 추가
```
데이터 구조 정의 → HTML 컨테이너 추가 → render 함수 구현 → 사이드바 메뉴 연결 → JS 검증 → 배포 → CLAUDE.md 동기화
```

## 기존 뷰 수정
```
현재 상태 Read → 수정 구현 → JS 검증 → 배포
                                        || CLAUDE.md 동기화 (구조 변경 시에만)
```

## 사이드바 메뉴 변경
```
HTML 수정 → JS 검증 → 배포 → CLAUDE.md 사이드바 섹션 동기화
```

## 산출물 문서 생성
```
프로토타입 현재 상태 분석 → 문서 작성 → 검토/수정 → mnt/Prototype/ 배포
```
