## 1. 커리큘럼 수정 이슈

### 문제
- 커리큘럼에 추가/삭제/수정이 일어나면 기존 콘텐츠에 영향 (스위칭 이슈)
- 커리큘럼만으로 제공한 케이스는 역추적이 어려움

### 개발팀 제안 방향
- 커리큘럼 변경 시 → **신규 커리큘럼으로 복제** 후 변경 내용 적용 (기존 것 수정 X)
- 커리큘럼 내부에 **고유 key 값** 필요 (역추적용)
- **워크테이블(temp) 방식**: 작업/수정은 임시 영역에서 진행 → 운영 배포 시점에 DB 업데이트
- 커리큘럼에도 **검수 → 배포 프로세스** 필요

### 합의된 정책
- 레이블 수정: 즉시 수용 가능
- 구조 변경(추가/삭제): 검수 요청 → 검수 → 배포 단계
- 서비스 중인 커리큘럼 수정: 동일 프로세스 적용

### 영향 범위
- 과목 바인딩 ↔ 콘텐츠 ↔ 커리큘럼 3자 관계

## 2. 커리큘럼 증분 업데이트 (배치 수신)

### 상황
- 뉴드림스에서 배치로 커리큘럼을 받아오는 경우 존재
- 예: 최초 1과정 10세트로 운영 중 → 2과정 20세트까지 증가
- 기존 콘텐츠는 그대로 운영 중이고, 커리큘럼만 확장됨

### 필요한 프로세스
- 시스템 연동을 통해 **업데이트 알람** → 콘텐츠팀이 준비할 수 있도록
- 기존 콘텐츠 영향 없이 신규 범위만 추가 작업

### 버저닝 재설계 필요
- **현재**: 콘텐츠 유형별 버저닝
- **필요**: **커리큘럼 단위 버저닝** + history 관리 워크플로우
- 커리큘럼이 변경될 때마다 버전이 올라가고, 어떤 시점의 커리큘럼으로 콘텐츠가 만들어졌는지 추적 가능해야 함

### 메모 1과의 연결
- 메모 1의 "워크테이블 → 운영 배포" 개념과 동일 선상
- 커리큘럼 복제 + 버저닝 + 검수/배포 프로세스가 하나의 흐름으로 통합되어야 함

---

## 연결되는 프로토타입 설계

- 커리큘럼 상태 게이트: draft → active (현재 설계)
- 여기에 **버전 관리 레이어** 추가 필요: v1(1과정 10세트) → v2(2과정 20세트) → ...
- 콘텐츠는 특정 커리큘럼 버전에 바인딩
- 배치 수신(외부 수신 현황 뷰)과 커리큘럼 버전 연동
