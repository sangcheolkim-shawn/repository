1 시스템 구조와 운영 원칙

CMS는 콘텐츠의 설계, 제작, 검수, 배포, 서비스를 일관된 파이프라인으로 관리합니다. 13개 메뉴는 5개 기능 영역으로 분류됩니다.

5개 기능 영역

영역포함 메뉴역할
① 기반 설정 입력양식 관리, 커리큘럼 관리, 시스템 관리 콘텐츠 구조(양식, 과정 계층)를 설계하고 시스템 환경을 구성합니다. 모든 작업의 전제 조건입니다.
② 콘텐츠 작업 작업 설정, 작업 진행 보드, 콘텐츠 등록 설계된 구조에 맞춰 실제 콘텐츠를 제작합니다. 작업 배정부터 콘텐츠 입력까지.
③ 품질 · 배포 검수 관리, 채점 시뮬레이션, 배포 관리 제작된 콘텐츠의 품질을 확인하고 서비스에 배포합니다.
④ 서비스 연결 과목코드 연결, 서비스 콘텐츠 관리 콘텐츠를 실제 서비스 과목코드에 매핑하고, 서비스 중인 현황을 모니터링합니다.
⑤ 외부 연동 커리큘럼 동기화, 외부 수신 현황 외부 시스템 변경을 감지하고, 수신된 데이터를 처리합니다.

운영 3원칙

📐
원칙 1 — 콘텐츠 독립성
콘텐츠는 특정 과목코드(서비스)와 완전히 독립적으로 존재합니다. 콘텐츠를 먼저 만들고, 별도의 '연결' 단계에서 과목코드와 매핑합니다. 이 덕분에 같은 콘텐츠를 여러 서비스에 재활용할 수 있습니다.
🏗
원칙 2 — 커리큘럼 우선
외부 시스템에서 변경 사항이 수신되어도, CMS 내부에서 작업 중인 커리큘럼 내용이 항상 우선합니다. 외부 변경을 자동으로 덮어쓰지 않으며, 반드시 운영자의 판단(반영/보류/반려)을 거칩니다.
📦
원칙 3 — 양식 = 전달 단위
시스템에 전달되는 최소 단위는 '양식(Form)'입니다. 동일한 학습 콘텐츠라도 대상 시스템(뉴드림스, 눈높이 등)마다 별도 양식으로 분리하여 전달합니다. 양식 레벨에서 과목코드가 독립적으로 연결됩니다.

2 콘텐츠 독립성 아키텍처

모든 콘텐츠는 아래 5단계 파이프라인을 거칩니다. 각 단계는 독립적이며, 순서가 바뀌면 안 됩니다.

① 양식 정의
입력양식 관리 → ② 구조 설계
커리큘럼 관리 → ③ 콘텐츠 제작
작업설정 → 등록 → ④ 검수(STG)
검수 관리 → ⑤ 배포(PRD)
배포 관리
⚡ 과목코드 연결 (독립)
시점 자유 · 연결/해제 가능
💡
왜 이 순서인가? 양식이 없으면 커리큘럼에 배치할 수 없고, 커리큘럼이 없으면 작업을 배정할 수 없습니다. 검수는 연결된 시스템의 STG에 자동 반영되고, 배포는 PRD로 확정합니다. 과목코드 연결은 독립 작업으로 시점에 제한 없이 언제든 연결/해제할 수 있습니다.

유형코드 자동 파생

양식을 커리큘럼에 배치하면 유형코드(typeCode)가 자동 생성됩니다. 유형코드는 레거시 시스템 호환을 위한 것으로, 운영자가 직접 관리하지 않습니다. UI에 노출되지 않으며 시스템이 내부적으로 처리합니다.

3 6대 운영 워크플로우

CMS에서 수행하는 모든 업무는 아래 6가지 워크플로우 중 하나에 해당합니다. 작업 시작 전에 "내 업무가 어떤 워크플로우인지"를 먼저 판단하세요.

1
신규 커리큘럼 구축
새 학습 과정이 필요할 때
입력양식 정의→ 커리큘럼 설계→ 작업 설정→ 작업보드 모니터링→ 콘텐츠 등록→ 검수(STG)→ 배포(PRD) 과목코드 연결 — 독립 · 시점 자유
시작 조건: 신규 학습 과정에 대한 기획 확정. 양식 구조와 커리큘럼 설계가 선행되어야 합니다.
진행 순서:
1. 입력양식 관리에서 필요한 양식(필드 구성)을 먼저 정의합니다. 기존 양식을 재활용할 수 있는지 확인합니다.
2. 커리큘럼 관리에서 과정→단원→세트 계층 구조를 설계하고, 양식을 배치합니다 (splitDepth 지정).
3. 커리큘럼을 '활성' 상태로 전환한 뒤, 작업 설정에서 양식별 담당자를 배정하고 일정을 설정합니다.
4. 배정된 작업이 작업 진행 보드에 자동 등록됩니다. 콘텐츠 담당자가 콘텐츠 등록에서 실제 콘텐츠를 입력합니다.
5. 입력 완료 건은 검수 관리에서 품질 확인 후 승인/반려합니다. 검수 진입 시 과목코드가 연결된 시스템의 STG에 자동 반영됩니다.
6. 승인된 콘텐츠를 배포 관리에서 PRD로 확정합니다. 과목코드가 연결된 시스템에 서비스가 자동 반영됩니다.
7. 과목코드 연결은 독립 작업입니다. 배포 전/후 언제든 연결/해제할 수 있으며, 배포된 콘텐츠에 나중에 코드를 연결하면 즉시 서비스에 반영됩니다.
종료 조건: 배포(PRD) 완료. 과목코드 연결은 별도 시점에 처리 가능.
기획자: 양식·커리큘럼 콘텐츠팀: 등록 검수팀: 검수·배포 운영팀: 코드연결
2
외부 변경 대응 (동기화)
커리큘럼 동기화에서 변경 감지 시
동기화 감지→ 영향 분석 · 의사결정→ 편집버전 생성→ 콘텐츠 수정→ 검수(STG) → 배포(PRD)
시작 조건: 커리큘럼 동기화 화면에서 syncStatus가 'recv-ahead'(검토필요) 또는 'diverged'(양쪽변경)인 항목 발생.
진행 순서:
1. 커리큘럼 동기화 대시보드에서 '조치필요' 건을 확인합니다.
2. 상세 뷰에서 변경 내용(diff)을 확인하고, 영향받는 시스템·과목코드를 분석합니다.
3. 의사결정 — 시스템별로 독립 판단합니다:
  • 반영: '편집버전 생성' → 커리큘럼 편집기로 이동하여 구조 반영
  • 보류: 추후 검토를 위해 보류 처리 (대시보드에 보류 상태 유지)
  • 반려: 수신 변경을 무시 (커리큘럼 우선 원칙)
4. 편집버전에서 구조 변경 후, 추가 콘텐츠가 필요하면 작업 설정 → 콘텐츠 등록으로 이어갑니다.
5. 변경된 콘텐츠를 검수(STG) → 배포(PRD)합니다.
종료 조건: 동기화 대시보드에서 해당 건의 syncStatus가 'matched'(일치) 또는 'cur-ahead'(배포가능)로 전환.
⚠️
핵심 의사결정 포인트: 동일 커리큘럼이 여러 시스템(ND/NH/SJ/NH365)에 연결되어 있으면, 시스템별로 반영/보류/반려를 각각 결정해야 합니다. 단, 커리큘럼 구조 자체는 하나이므로 편집버전은 공유됩니다.
3
내부 개정
기획자가 구조 변경을 결정할 때
커리큘럼 편집버전 생성→ 구조 수정→ 작업설정 → 콘텐츠등록→ 검수(STG) → 배포(PRD)
시작 조건: 기획자가 기존 커리큘럼의 구조 변경(단원 추가/삭제, 양식 교체 등)을 결정.
WF-1과의 차이: 양식 정의 단계를 건너뛰고, 기존 커리큘럼의 편집버전(Temp)을 생성하여 시작합니다. 이미 정의된 양식을 재활용합니다.
진행 순서:
1. 커리큘럼 관리에서 기존 활성 커리큘럼의 '편집버전 생성' 버튼을 클릭합니다.
2. 편집기에서 과정/단원/세트 구조를 수정합니다.
3. 변경으로 인해 추가 콘텐츠가 필요하면 작업 설정에서 새 작업을 배정합니다.
4. 나머지는 WF-1의 ③~⑤ 단계와 동일합니다.
종료 조건: 변경된 콘텐츠가 배포되고, 편집버전이 활성 버전으로 승격.
4
콘텐츠 수정 (유지보수)
오류 발견, 내용 업데이트 필요 시
작업보드에서 이슈 확인→ 콘텐츠등록에서 수정→ 검수(STG) → 배포(PRD)
시작 조건: 운영 중 콘텐츠 오류 발견, 교과 내용 업데이트, 사용자 피드백 반영 등.
특징: 가장 빈번하게 발생하는 워크플로우입니다. 커리큘럼 구조는 변경하지 않고, 콘텐츠(문제, 해설, 이미지 등)만 수정합니다.
진행 순서:
1. 작업 진행 보드에서 수정 대상 카드를 확인하거나, 직접 콘텐츠 등록으로 진입합니다.
2. 해당 콘텐츠를 수정한 뒤 '등록 완료' 처리합니다.
3. 검수 관리에서 수정 내용을 확인 후 승인합니다 (연결된 시스템 STG에 자동 반영).
4. 배포 관리에서 PRD로 확정합니다 (연결된 시스템 서비스에 자동 반영).
종료 조건: 수정 콘텐츠 배포(PRD) 완료.
5
과목코드 연결 관리
과목코드 연결/해제 — 시점 자유, 독립 작업
코드연결 설정→ 서비스콘텐츠 확인
시작 조건: 새로운 과목코드 추가, 기존 콘텐츠를 새 시스템에 연결, 또는 기존 연결 해제가 필요할 때.
특징: 검수·배포와 완전히 독립된 작업입니다. 콘텐츠 변경 없이 시스템 연동 채널만 설정/해제합니다. 시점 제한 없이 언제든 실행 가능합니다.
진행 순서:
1. 과목코드 연결에서 대상 커리큘럼을 찾고, 양식별 과목코드를 연결하거나 해제합니다.
2. 서비스 콘텐츠 관리에서 연결 결과가 정상인지 확인합니다.
3. 배포(PRD) 완료된 콘텐츠에 코드를 연결하면, 해당 시스템 서비스에 즉시 반영됩니다.
종료 조건: 서비스 콘텐츠 관리에서 과목코드 매핑 정상 확인.
6
외부 데이터 수신 처리
외부 시스템에서 데이터 일괄 수신 시
외부수신 확인→ 구조 변경 vs 콘텐츠만→ 커리큘럼 or 콘텐츠등록→ 검수(STG) → 배포(PRD)
시작 조건: 외부 시스템에서 데이터가 일괄 수신될 때.
의사결정 포인트: 수신 데이터가 구조 변경(과정/단원/세트 변경)을 포함하면 WF-2(동기화)로 이어지고, 콘텐츠만(문제/해설 데이터)이면 WF-4(유지보수)로 이어집니다.
진행 순서:
1. 외부 수신 현황에서 수신된 데이터의 유형과 상태를 확인합니다.
2. 구조 변경이 포함된 경우: 커리큘럼 동기화에서 변경 사항을 분석한 뒤 WF-2를 따릅니다.
3. 콘텐츠만인 경우: 콘텐츠 등록에서 직접 반영하고 WF-4를 따릅니다.
종료 조건: 수신 데이터의 처리 상태가 '완료'로 전환.

4 메뉴별 운영 지침

각 메뉴의 역할, 진입 시점, 수행 업무, 다음 단계를 정의합니다. "이 메뉴에서 뭘 해야 하지?"에 대한 답입니다.

기반 설정

기반 입력양식 관리 form-manage
역할콘텐츠의 입력 구조(필드 구성)를 정의합니다. 하나의 양식은 하나의 시스템 전달 단위입니다.
진입 시점WF-1 첫 단계 / 새로운 콘텐츠 유형이 필요할 때 / 기존 양식 필드를 수정해야 할 때
수행 업무양식 생성·편집, 필드 추가·삭제·순서변경, 대상 시스템(ND/NH/SJ/NH365) 설정, 필드 유형·규칙 설정

의사결정 포인트 2: 편집버전 생성 시점

상황판단
커리큘럼 구조 변경이 필요 (단원 추가/삭제, 양식 교체) 반드시 편집버전 생성 후 수정. 활성 커리큘럼 직접 수정 금지.

의사결정 포인트 3: 배포 우선순위

상황판단
긴급 오류 수정 (서비스 영향) 즉시 배포. 검수 간소화 가능 (긴급 승인).
정기 업데이트 (교과 개정, 신규 콘텐츠) 예약 배포. 서비스 영향 최소화 시간대 선택.
다중 시스템 배포 (ND/NH/SJ에 동시 영향) 시스템별 순차 배포 권장. ND 먼저 → 확인 → NH → SJ.

6 다중 시스템 운영

하나의 커리큘럼이 ND(뉴드림스), NH(눈높이), SJ(성장판), NH365(눈높이365)에 각각 다르게 연결될 수 있습니다.

대상 시스템

● ND 뉴드림스 ● NH 눈높이 ● SJ 성장판 ● NH365 눈높이365

다중 시스템 운영 원칙

📐
원칙 A — 커리큘럼은 시스템에 독립
"수학 기초 과정"은 어느 시스템에서 쓰이든 동일한 구조입니다. 시스템별 차이는 양식 레벨의 과목코드 연결에서 발생합니다. 커리큘럼을 시스템별로 복제하지 않습니다.
🔀
원칙 B — 같은 커리큘럼, 다른 양식 조합
ND는 [기본양식+심화양식], NH는 [기본양식만], SJ는 [기본양식+특별양식]을 사용할 수 있습니다. 양식별로 과목코드가 다르게 연결되므로, 시스템별 콘텐츠 범위가 자연스럽게 달라집니다.
⚡
원칙 C — 동기화 시 시스템별 영향 분석 필수
외부 변경 감지 시, 동일 커리큘럼에 연결된 모든 시스템의 영향을 각각 확인합니다. ND는 반영하되 NH는 보류하는 것이 가능합니다. 단, 편집버전은 공유됩니다.
🎯
원칙 D — 배포는 양식×과목코드 단위
커리큘럼 전체가 한 번에 배포되지 않습니다. 양식별, 과목코드별로 독립 배포됩니다. "ND의 수학 기본양식만 먼저 배포하고, NH는 다음 주에 배포"가 가능합니다.

시나리오 예시: "수학 기초 과정" 구조 변경 감지

1단계 — 감지: 커리큘럼 동기화에서 "과정 8→9, 세트 80→90" 변경 감지
2단계 — 분석: 영향 범위 확인 — ND(2건), NH(1건), SJ(1건) 총 4개 과목코드에 영향
3단계 — 의사결정:
  • ND·SJ → 동일 구조 사용 → 반영 결정 (편집버전 생성)
  • NH → 이미 자체 수정 완료 → 수신 무시 (커리큘럼 우선 원칙)
4단계 — 작업: 편집버전에서 구조 수정 → 콘텐츠팀에 추가 세트 10건 작업 배정
5단계 — 배포: 작업 완료 → 검수 → ND 먼저 배포, SJ는 다음 주 배포
6단계 — 확인: 동기화 대시보드 자동 갱신 — ND·SJ = "일치(matched)", NH = "배포가능(cur-ahead)"

7 동기화 상태(syncStatus) 관리

커리큘럼 동기화 대시보드의 5가지 상태와 각 상태에서의 조치 방법입니다.

상태의미조치 필요 여부가능한 액션
matched 일치 CMS 커리큘럼과 외부 구조가 동일 조치 불필요 ✅ 상세 보기 (확인 용도)
cur-ahead 배포가능 CMS 커리큘럼이 외부보다 앞서 있음 (CMS에서 먼저 수정) 배포 확인 권장 배포 확인, 상세 보기
recv-ahead 검토필요 외부에서 변경이 감지됨 (외부가 앞서감) 반드시 조치 ⚠️ 비교 보기 → 반영(편집버전) / 보류 / 반려
diverged 양쪽변경 CMS와 외부 모두 변경됨 (충돌 가능) 반드시 조치 🔴 양쪽 비교 → 커리큘럼 기준 배포 / 편집버전에서 병합
unlinked 미연결 외부 수신 데이터와 매칭되는 CMS 커리큘럼 없음 확인 필요 커리큘럼 연결 설정 / 무시 처리
💡
구조 지문(Structural Fingerprint): 동기화 상태는 버전 번호가 아닌 '구조 지문' — 커리큘럼 구조를 해싱한 값 — 으로 판정합니다. 따라서 버전이 달라도 구조가 동일하면 'matched'로 표시됩니다.

8 용어 사전

CMS 운영에서 사용하는 핵심 용어를 정의합니다.

양식 (Form) 콘텐츠의 입력 구조를 정의한 템플릿. 시스템에 전달되는 최소 단위이며, 대상 시스템별로 별도 양식을 사용합니다. 하나의 양식은 여러 필드(텍스트, 이미지, 선택지 등)로 구성됩니다.
커리큘럼 학습 과정의 계층 구조. 과정→단원→세트로 구성되며, 각 세트에 양식이 배치됩니다. 상태: 초안(draft) → 활성(active). 수정이 필요하면 편집버전(temp)을 생성합니다.
편집버전 (Temp) 활성 커리큘럼의 수정 작업본. 서비스 중인 활성 버전을 직접 수정하지 않고, 편집버전을 만들어 수정 → 검수 → 배포 → 활성으로 승격하는 안전한 변경 프로세스입니다. 커뮤니케이션에서는 "편집버전"으로 통일합니다.
과목코드 대상 시스템(ND/NH/SJ/NH365)에서 콘텐츠를 식별하는 코드. 하나의 양식에 여러 과목코드가 연결될 수 있으며, 이를 통해 동일 콘텐츠를 여러 서비스에 매핑합니다.
동기화 (Sync) CMS 내부 커리큘럼 구조와 외부 시스템의 구조를 비교하여 차이를 감지하는 프로세스. 구조 지문(Structural Fingerprint)을 사용합니다.
syncStatus 동기화 상태를 나타내는 5가지 값: matched(일치), cur-ahead(배포가능), recv-ahead(검토필요), diverged(양쪽변경), unlinked(미연결).
콘텐츠 독립성 콘텐츠가 특정 과목코드(서비스)와 독립적으로 존재하는 아키텍처 원칙. 콘텐츠를 먼저 만들고, '연결' 단계에서 과목코드와 매핑합니다.
splitDepth 커리큘럼 구조에서 양식이 배치되는 depth 레벨. 예를 들어 splitDepth=3이면 3번째 계층(세트)에 양식이 연결됩니다.
유형코드 (typeCode) 양식 선택 시 자동으로 파생되는 레거시 호환 코드. UI에 노출되지 않으며 시스템이 내부적으로 관리합니다.
