# CLAUDE.md 작성 가이드 — 주니어 기획자용

> **이 문서의 목적**: 대교 CMS 고도화 프로젝트에서 실제로 운영한 CLAUDE.md 사례를 바탕으로, AI 에이전트와 협업할 때 "어떤 정보를, 어떤 구조로, 왜 적어야 하는지"를 설명합니다. 화면 기획, UI 기획, 프로젝트 관리 관점에 집중합니다.

---

## CLAUDE.md가 뭔가요?

CLAUDE.md는 AI 에이전트에게 "이 프로젝트에서 지켜야 할 규칙과 맥락"을 알려주는 문서입니다. 사람으로 치면 프로젝트에 새로 합류한 팀원에게 건네는 온보딩 문서와 같습니다.

AI는 대화가 길어지면 앞부분의 맥락을 잊을 수 있는데, CLAUDE.md는 매 세션마다 자동으로 읽히기 때문에 "늘 기억하게 해야 할 것"을 적어두는 용도입니다. 잘 쓰면 매번 같은 설명을 반복하지 않아도 되고, AI가 프로젝트 맥락에 맞는 결과물을 훨씬 빠르게 만들어냅니다.

---

## 이 프로젝트 CLAUDE.md의 전체 구조

실제 CMS 프로젝트의 CLAUDE.md는 아래 섹션들로 구성되어 있습니다. 각 섹션이 "왜 필요한지"를 기획 관점에서 설명합니다.

```
1. 프로젝트 개요           ← 프로젝트의 한 줄 정의
2. 핵심 파일 구조           ← 산출물이 어디에 있는지
3. 워크플로우 & 작업 관리    ← 일하는 방식의 규칙
4. 화면 구조 (뷰 & 네비게이션) ← 메뉴 구조와 화면 흐름
5. 업무 프로세스 흐름        ← 실제 사용자가 밟는 단계
6. 핵심 데이터 구조          ← 화면에 표시되는 데이터의 모양
7. 산출물 체계              ← 어떤 문서를 언제 만드는지
8. 변경 이력               ← 뭘 바꿨고 왜 바꿨는지
9. 주의사항 & 알려진 이슈    ← 실수하기 쉬운 함정들
```

아래부터 기획자에게 특히 중요한 섹션을 하나씩 설명합니다.

## 1. 프로젝트 개요 — "한 줄로 말할 수 있어야 한다"

### 실제 사례

```markdown
대교(Daekyo) 교육 콘텐츠 관리 시스템(CMS) 고도화 프로젝트.
단일 HTML 파일로 구현된 인터랙티브 프로토타입이며, 실제 개발 전 UX/워크플로우 검증 목적.

- 담당자: Shawn (sangcheol.kim@daekyo.co.kr)
- 현재 버전: cms_stg_prd_v8.html
```

### 왜 중요한가

AI가 "이건 실서비스 코드가 아니라 프로토타입"이라는 것을 모르면, 불필요하게 에러 핸들링이나 보안 처리를 넣으려 합니다. 프로젝트의 목적과 범위를 명확히 알려줘야 AI가 적절한 수준의 결과물을 만듭니다.

### 기획자가 챙길 포인트

프로젝트 개요에 꼭 포함해야 할 것:

- **프로젝트 이름과 조직** — AI가 용어를 맥락에 맞게 사용하려면 필요
- **목적** — "검증용 프로토타입" vs "운영 서비스"는 결과물의 깊이가 완전히 다름
- **담당자 연락처** — AI가 문서 안에 담당자 정보를 자연스럽게 반영
- **현재 버전** — 어떤 파일이 "진짜"인지 혼동 방지

## 2. 파일 구조 — "산출물 지도"

### 실제 사례

```
mnt/Prototype/
├── cms_stg_prd_v8.html         ← 최신 배포본
├── CMS_워크플로우_설명서_v1.docx
├── CMS_콘텐츠독립성_아키텍처_정책문서_v1.docx
├── tasks/
│   ├── workflow.md             ← 일하는 방식 규칙
│   ├── todo.md                 ← 작업 추적
│   └── lessons.md              ← 교훈 기록
└── archive/                    ← 이전 버전 보관
```

### 왜 중요한가

AI에게 "보고서 만들어줘"라고 했을 때, 어디에 저장해야 하는지, 기존에 비슷한 문서가 있는지를 모르면 중복 파일이 생기거나 엉뚱한 위치에 저장합니다. 파일 구조를 알려주면 AI가 기존 산출물을 참조하고, 올바른 위치에 새 파일을 만듭니다.

### 주니어 기획자가 챙길 포인트

- **최신 파일에 화살표(←) 표시**: 여러 버전이 있을 때 "지금 이것이 진짜"를 명시
- **archive 폴더 운영**: 구버전은 버리지 말고 보관 — AI가 "v7에서 어떻게 했었지?" 비교 참조 가능
- **tasks/ 폴더 분리**: 작업 관리 문서와 산출물을 분리하면 AI가 "할 일 목록"과 "결과물"을 혼동하지 않음

## 3. 워크플로우 & 작업 관리 — "AI와 일하는 방식의 규칙"

이 프로젝트에서 가장 효과가 컸던 부분입니다. AI에게 "무엇을 만들어라"만 말하는 것이 아니라 "어떻게 일해라"를 정의해둔 것입니다.

### 3-1. 계획 먼저 (Plan Before Action)

#### 실제 사례

```markdown
### 핵심 규칙
1. 계획 먼저 — 3단계 이상 작업은 계획 모드로 시작, tasks/todo.md에 기록
4. 완료 전 검증 — "시니어 엔지니어가 승인할까?" 자문 후 완료 표시
```

#### 기획자에게 이것이 왜 중요한가

AI에게 복잡한 작업을 시키면 바로 실행부터 하려는 경향이 있습니다. "일단 해보고 고치자"가 아니라 "계획을 세우고, 그 계획을 todo.md에 기록한 뒤, 하나씩 체크하면서 진행해라"라고 규칙을 잡아두면 결과물의 품질이 확연히 달라집니다.

기획서 작성에 적용하면: "기획서 써줘" 대신 "목차를 먼저 잡고, 각 섹션을 순서대로 채우되, 한 섹션 끝날 때마다 체크해라"가 됩니다.

### 3-2. 자기개선 루프 (Lessons Learned)

#### 실제 사례 — lessons.md

```markdown
### L2: Confluence 기획서 — 코드 중심 vs 비즈니스 로직 중심
- 상황: [기능] 페이지들을 뷰 ID, 함수명, 상태 객체 중심으로 작성
- 문제: "커뮤니케이션 할 때 논리적으로 정리가 안됨"
- 수정: 기능목적 → 선행조건 → 화면구성 → 업무흐름 → 비즈니스규칙 구조로 재작성
- 규칙: 기획서는 항상 "왜/누가/어떤 순서로"를 먼저, 기술 세부사항은 뒤로
```

```markdown
### L9: "동선 정리" 요청을 메뉴 재구조화로 과도 확장하지 말 것
- 상황: "페이지 동선 찾기가 어렵다" 피드백에 사이드바 4그룹 재편까지 적용
- 문제: 사용자 의도는 "페이지별 디테일 보강"이었고, 메뉴 구조 변경은 불필요
- 규칙: 피드백의 범위를 확장 해석하지 말 것. 가장 작은 단위부터 적용.
```

#### 기획자에게 이것이 왜 중요한가

AI에게 수정 피드백을 줬을 때, 같은 실수를 다음 세션에서 또 반복하는 경우가 많습니다. lessons.md에 "상황 → 문제 → 수정 → 규칙" 4줄 형식으로 기록해두면, AI가 매 세션 시작 시 이 파일을 읽고 같은 실수를 피합니다.

특히 L2와 L9는 기획자에게 직접적으로 유용한 교훈입니다:
- **L2**: 기획서는 기술 용어가 아니라 업무 흐름 중심으로 써야 소통이 됨
- **L9**: 피드백 해석 범위를 최소한으로 잡아야 불필요한 작업을 방지

### 3-3. todo.md — 작업 추적

#### 실제 사례

```markdown
## 진행 중
(현재 진행 중인 작업 없음)

## 대기 중 / 향후 작업
- [ ] Step Indicator를 검수·배포 구간 뷰에도 주입
- [ ] 프로토타입 신규 뷰 개발 (batch-receive 등)
- [ ] Confluence 정책 문서 정비

## 완료된 작업
### 2026-04-20 — v9 커리큘럼+양식 구조 분리
- [x] 아키텍처 설계: typeNodes → forms 배열 분리 결정
- [x] CURRICULUM_DATA: depths에서 typeNodes 제거
- [x] JS 문법 검증 통과 + 배포 + CLAUDE.md 동기화
```

#### 기획자에게 이것이 왜 중요한가

3섹션(진행/대기/완료) 구조가 핵심입니다. AI가 "지금 뭘 해야 하는지"를 한눈에 파악할 수 있고, 완료 이력을 보면서 "이전에 어떤 방식으로 작업했는지" 패턴을 학습합니다.

기획 프로젝트에서는 이렇게 활용할 수 있습니다:
- **진행 중**: "기획서 초안 작성 — 화면구성 섹션까지 완료"
- **대기 중**: "디자이너 리뷰 후 수정 반영", "개발팀 확인 필요한 제약조건 정리"
- **완료**: 날짜와 함께 "무엇을 결정했는지" 기록 (나중에 의사결정 히스토리가 됨)

## 4. 화면 구조 (뷰 & 네비게이션) — "화면 기획의 핵심"

### 실제 사례 — 사이드바 메뉴 구조

```markdown
작업 진행 보드              [칸반 보드, 최상위 메뉴]

콘텐츠 작업
├── 작업 설정               [커리큘럼 단위 작업 배정]
└── 콘텐츠 등록             [실제 콘텐츠 등록 작업]

검수 & 배포
├── 검수 관리
└── 배포 관리               [바인딩 완료 콘텐츠만 배포]

서비스 연결 관리
├── 과목코드 바인딩         [커리큘럼↔과목코드 매핑]
└── 서비스 콘텐츠 관리      [배포 완료 콘텐츠 관리]

양식 관리
├── 입력 양식 관리          [필드 양식 정의]
└── 커리큘럼 관리           [depth·양식 매핑, 트리 에디터]

설정
└── 시스템 관리
```

### 왜 이 형식이 좋은가

화면 구조를 텍스트로 적을 때 이 프로젝트의 방식이 우수한 이유가 세 가지 있습니다.

**첫째, 트리 구조(├── └──)로 그룹-하위 관계가 한눈에 보입니다.** Figma나 PPT에서 메뉴를 시각적으로 그리면 예쁘지만, AI에게 전달하거나 Confluence에 붙여넣을 때는 텍스트 트리가 훨씬 명확합니다. 이 형식을 그대로 기획서에 활용할 수 있습니다.

**둘째, 각 메뉴 옆에 [한 줄 설명]이 있습니다.** "배포 관리"만 쓰면 AI가 이 화면이 뭘 하는 건지 추측해야 하지만, "[바인딩 완료 콘텐츠만 배포]"를 붙이면 비즈니스 규칙까지 전달됩니다.

**셋째, 그룹 이름이 "동사+명사" 패턴입니다.** "콘텐츠 작업", "검수 & 배포", "서비스 연결 관리" — 사용자가 어떤 "행위"를 하는 영역인지가 그룹 이름에서 바로 드러납니다.

### 실제 사례 — 페이지 간 동선 설계

```markdown
페이지별 디테일: 각 뷰 상단에 워크플로 Step Indicator 브레드크럼,
하단에 이전/다음 단계 CTA 렌더링.
사이드바 구조와는 독립적인 페이지 내 동선 보조 UI.
양식 배지/커리큘럼명 클릭 시 관련 뷰로 점프.
```

### 기획자가 배울 점

이 프로젝트에서는 화면 동선을 두 가지 층위로 분리했습니다:
- **사이드바**: 전체 메뉴 구조 (기능 그룹 기반, 안정적으로 유지)
- **페이지 내 Step Indicator**: 실제 업무 순서를 안내하는 보조 UI

처음에는 사이드바 자체를 업무 순서대로 재정렬했지만, "사용자가 매일 보는 앵커"인 사이드바를 자주 바꾸면 혼란을 준다는 교훈(L9)을 얻고, 사이드바는 그대로 두고 페이지 안에서 "지금 여기예요, 다음은 저기예요"를 안내하는 방식으로 전환했습니다.

이것은 기획 원칙으로 기억할 만합니다: **글로벌 네비게이션(사이드바)은 안정적으로 유지하고, 업무 흐름 안내는 로컬 UI(브레드크럼, CTA 버튼)로 분리하라.**

## 5. 업무 프로세스 흐름 — "사용자가 밟는 단계"

### 실제 사례

```markdown
① 입력 양식 정의 → ② 커리큘럼 설계 → ③ 콘텐츠 제작 → ④ 과목코드 바인딩 → ⑤ 배포
```

이 한 줄이 전체 시스템의 업무 흐름을 요약합니다. 그리고 각 단계마다 어떤 화면에서 어떤 작업을 하는지를 매핑합니다:

```markdown
- ① 입력 양식: 양식 관리 > 입력 양식 관리
- ② 커리큘럼: 양식 관리 > 커리큘럼 관리 (depth 구조 + 양식 매핑)
- ③ 콘텐츠 제작: 작업 설정 → 콘텐츠 등록 (커리큘럼 단위 작업)
- ④ 과목코드 바인딩: 서비스 연결 관리 > 과목코드 바인딩
- ⑤ 배포: 검수 → 배포 관리 (바인딩 완료 콘텐츠만 대상)
```

### 기획자에게 이것이 왜 중요한가

AI에게 "콘텐츠 등록 화면을 기획해줘"라고 하면, 이 흐름을 알고 있는 AI는 "③번 단계니까, ②에서 만든 커리큘럼을 전제로 화면을 구성해야겠다"고 판단합니다. 프로세스 전후 맥락을 모르면 동떨어진 화면이 나옵니다.

기획서에 이 구조를 활용하는 방법:
- **전체 흐름을 한 줄로 먼저 쓰고**, 각 단계를 화면과 매핑
- **선행 조건을 명시**: "④ 바인딩은 ③ 콘텐츠 제작이 완료된 후에만 가능"
- **흐름에 없는 화면은 설명 추가**: "시스템 관리는 프로세스 흐름과 무관한 관리자 전용 화면"

## 6. 화면 카드(UI 컴포넌트) 정의 — "화면에 뭐가 보이는지"

### 실제 사례 — Work-config 카드

```
┌─ 수학 기초 과정                     활성 ─┐
│  과목 1 › 과정 8 › [세트 80] › 학습일 400  │
│  바인딩 코드: [M 수학]                      │
│  입력 양식: [정답·해설] [자동채점]           │
│  전체 진행률: ███████░░░ 72%               │
│  ─────────────────────────────────        │
│  ■ 홍길동  정답·해설  1~40   ████░░ 70%    │
│  ■ 김철수  정답·해설  41~80  ██████ 100%   │
└──────────────────────────────────────────┘
```

### 왜 이 방식이 효과적인가

Figma 시안이 아직 없는 단계에서 화면 레이아웃을 AI에게 전달하는 가장 빠른 방법이 이런 "텍스트 와이어프레임"입니다. 이 사례가 좋은 이유:

- **실제 데이터**로 그렸습니다. "제목", "설명", "진행률" 같은 자리표시자가 아니라 "수학 기초 과정", "홍길동", "72%"처럼 구체적인 값을 넣었기 때문에 AI가 데이터 바인딩까지 이해합니다.
- **계층 구조**가 드러납니다. 카드 안에 요약 영역과 상세(담당자별) 영역이 구분선으로 나뉘어 있습니다.
- **상태 표현**이 포함됩니다. "[활성]", "████░░ 70%" 같은 상태값이 어디에 어떻게 보여야 하는지가 명확합니다.

### 기획자가 활용하는 방법

새 화면을 기획할 때 이 형식을 CLAUDE.md에 넣어두면, AI가 프로토타입을 만들 때 이 레이아웃을 참조합니다:

```
┌─ [화면명]                                    ─┐
│  [첫 번째 줄: 핵심 정보]                        │
│  [두 번째 줄: 보조 정보 / 태그]                  │
│  [세 번째 줄: 상태 표시]                         │
│  ────────────────────────────                  │
│  [상세 목록 / 하위 항목]                         │
└─────────────────────────────────────────────┘
```

## 7. 칸반 보드 — 상태 흐름과 카드 구성 요소

### 실제 사례

```markdown
### 페이즈 구조
assign(담당 배정) → register(콘텐츠 등록) → review(검수) → deploy(배포)

### 카드 표시 요소
1. 시스템 배지 (ND/NH/SJ)
2. 유형코드 (ANS/CBT/FLC/STDG...)
3. 우선순위 (긴급 시 빨간 뱃지)
4. 소스 배지 (외부 연동 파란색 / 직접 등록 초록색)
5. 템플릿명
6. 과목코드 + 과정명
7. 진행률 바 (세트 기준)
8. 담당자 + 액션 버튼
```

### 기획자가 배울 점

칸반 화면을 기획할 때 두 가지를 CLAUDE.md에 정의해야 합니다:

**첫째, 상태 흐름**: 카드가 왼쪽에서 오른쪽으로 어떤 단계를 거치는지. "assign → register → review → deploy"처럼 영문 키와 한글 라벨을 같이 적어두면 AI가 데이터 구조와 UI 라벨을 모두 이해합니다.

**둘째, 카드 구성 요소를 번호로 정리**: 카드 안에 "뭐가 보여야 하는지"를 우선순위 순으로 나열합니다. AI는 이 순서대로 레이아웃을 배치합니다. 나중에 "4번(소스 배지)은 빼자"처럼 번호로 소통할 수 있어서 편합니다.

## 8. 산출물 체계 — "뭘 언제 만드는가"

### 실제 사례

```markdown
| 순서 | 파일 | 용도 | 생성 시점 |
|------|------|------|-----------|
| 00 | cms_stg_prd_v8.html | 인터랙티브 프로토타입 | 매 작업 완료 시 |
| 01 | CMS_워크플로우_설명서_v1.docx | 전체 업무 프로세스 설명서 | 프로세스 확정 후 |
| 02 | CMS_콘텐츠독립성_정책문서_v1.docx | 아키텍처 정책 (Confluence용) | 아키텍처 확정 후 |

### 산출물 버전 규칙
- 프로토타입: cms_stg_prd_v{N}.html — 메이저 변경 시 N 증가
- 문서: {문서명}_v{N}.docx — 내용 확정 시 N 증가
- 프로토타입이 변경되면 관련 문서도 동기화 필요 여부 점검
```

### 기획자에게 이것이 왜 중요한가

AI에게 "문서 만들어줘"라고 할 때, 산출물 목록이 있으면 AI가 "이미 01번 워크플로우 설명서가 있으니, 이걸 업데이트할까요, 새로 만들까요?"라고 물어볼 수 있습니다.

특히 **"프로토타입이 변경되면 관련 문서도 동기화 필요 여부 점검"** 규칙이 중요합니다. 프로토타입에서 화면 구조가 바뀌었는데 기획서는 구버전 그대로인 상황을 방지합니다.

## 9. 변경 이력 — "의사결정 히스토리"

### 실제 사례 (기획 관련 발췌)

```markdown
10. Work-config 카드 간소화 (v6)
    — 복잡한 다행 구조를 심플 카드로 대체
13. v7 콘텐츠 독립성 완성
    — 과목코드 의존성 제거, 커리큘럼 중심 워크플로우 전환
23. 사이드바 활동 순서 재정렬 (v7.1, 롤백)
    — 사용자 피드백으로 기능 그룹 구조로 복원
    — 메뉴 재정렬은 불필요, 페이지별 디테일만 유지
30. 커리큘럼 관리 3-Screen UX (v8.1)
    — 리스트 뷰 + 스마트 빌더 + 트리 편집기 3화면 분리
```

### 기획자에게 이것이 왜 중요한가

변경 이력은 단순한 로그가 아니라 "의사결정 근거"입니다. 23번처럼 "시도했다가 롤백한 기록"이 특히 가치 있습니다. 새 세션에서 AI가 "사이드바를 활동 순서로 바꿔볼까요?"라고 제안하는 것을 미리 방지합니다.

기획서에서 이렇게 활용합니다:
- **번호를 매겨서** 이후 논의에서 "23번 결정 참조"처럼 빠르게 참조
- **롤백 사유**를 기록 — "왜 안 했는지"가 "왜 했는지"만큼 중요
- **버전(v6, v7, v8)**과 연결 — 어떤 버전에서 어떤 변화가 있었는지 추적 가능

## 10. 보조 문서 체계 — "tasks/ 폴더의 힘"

이 프로젝트에서 CLAUDE.md 하나로 끝내지 않고 tasks/ 폴더에 3개 파일을 분리한 것이 핵심 전략입니다.

### tasks/workflow.md — 일하는 방식

계획 → 실행 → 검증의 전체 프레임을 정의합니다. AI에게 "일의 순서"를 알려주는 문서입니다.

기획자용 활용 예시:
```markdown
## 기획서 작성 워크플로우
1. 목차 작성 → 리뷰 → 섹션별 내용 채우기 → 전체 검토
2. 화면 기획 시: 전체 흐름 먼저 → 개별 화면 → 상태/예외 케이스
3. 완료 전 체크: "이 기획서로 개발자가 바로 작업할 수 있는가?"
```

### tasks/todo.md — 작업 추적

3섹션(진행/대기/완료) 구조로 AI와 사람이 동일한 작업 상태를 공유합니다.

### tasks/lessons.md — 교훈 기록

"상황 → 문제 → 수정 → 규칙" 4줄 형식으로 실수 패턴을 누적합니다. 세션이 바뀌어도 같은 실수를 반복하지 않게 하는 핵심 장치입니다.

기획자용 활용 예시:
```markdown
### L1: 기획서에서 "~해야 한다"와 "~할 수 있다"를 구분하지 않음
- 상황: 배포 관리 기획서에서 "관리자가 스케줄을 생성할 수 있다"로 기술
- 문제: 개발팀이 선택 기능으로 해석 — 실제로는 필수 기능이었음
- 규칙: 필수 기능은 "~한다(SHALL)", 선택 기능은 "~할 수 있다(MAY)"로 명확히 구분
```

## 정리 — CLAUDE.md 체크리스트

새 프로젝트에서 CLAUDE.md를 처음 만들 때, 아래 항목을 하나씩 채워가면 됩니다. 모든 항목을 한 번에 완성할 필요는 없고, 프로젝트가 진행되면서 점진적으로 보강합니다.

**반드시 포함 (프로젝트 시작일에)**
- 프로젝트 이름, 목적, 담당자
- 현재 파일 구조 (어디가 "진짜"인지)
- 전체 업무 프로세스 한 줄 요약

**초기에 추가 (1주 이내)**
- 화면 구조 (메뉴 트리 + 한 줄 설명)
- 산출물 목록 (파일명, 용도, 생성 시점)
- tasks/ 폴더 생성 (workflow.md, todo.md, lessons.md)

**점진적 보강 (프로젝트 진행 중)**
- 주요 화면별 텍스트 와이어프레임
- 변경 이력 (번호 매겨서 누적)
- 교훈 기록 (피드백 받을 때마다)
- 용어 정의 (팀 내에서 혼동되는 단어가 나올 때마다)

**대교 CMS 프로젝트의 실제 규모 참고**: 이 프로젝트의 CLAUDE.md는 약 630줄이며, 보조 문서(workflow.md, todo.md, lessons.md)까지 합치면 약 900줄입니다. 3주간 9개 메이저 버전(v1~v9)을 거치면서 점진적으로 이 분량이 되었습니다.
