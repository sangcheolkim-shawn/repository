CORE CONCEPT

콘텐츠 독립성 아키텍처란?

콘텐츠(커리큘럼)가 외부 시스템의 과목코드에 종속되지 않고 독립적으로 존재하며, 바인딩(연결)을 통해 여러 시스템에 제공되는 구조입니다.

✕ 기존 (Legacy)

과목코드 M (수학)
  ├─ 과정 8개
  │  ├─ 세트 80개
  │  └─ 유형: ANS, CBT... (과목에 종속)
  └─ ND 시스템 전용 (1:1 고정)
⚠
과목코드가 바뀌면 콘텐츠 전체 재작업 필요. 타사 제공 불가.

✓ 신규 (v9)

커리큘럼: 수학 기초 과정
  ├─ Depth: 커리큘럼명 › 과정 › 세트 › 콘텐츠
  ├─ 양식: 해답지, 자동채점, 핵심점검
  ├─ 바인딩 → ND M (수학)
  └─ 바인딩 → SJ M (수학)
✓
하나의 커리큘럼을 N개 시스템 × N개 과목코드에 바인딩 가능.

Phase 1 — 양식 마스터 데이터 생성 (Day 1~3)

1

기존 CONTENT_TYPES → ncm_form_templates 매핑

기존 19개 유형코드를 양식 템플릿으로 변환합니다. 유형코드는 양식의 type_code 컬럼으로 그대로 이관됩니다.

💡
핵심 원리: 1 유형코드 → N 양식
양식은 과목·시스템 조합마다 여러 개 생길 수 있지만, 유형코드(ANS, CBT 등)는 변하지 않습니다. 양식이 100개든 1000개든 유형코드는 기존 레거시 그대로 승계됩니다.

✓ 실제 구조

유형코드 ANS는 1개로 고정.
양식(Form)이 과목·시스템별로 N개 생성되고,
각 양식이 type_code = 'ANS'를 참조.
유형코드 = 분류 기준, 양식 = 실제 입력 틀.
-- Phase 1: 유형코드 → 양식 템플릿 마이그레이션 SQL -- 기존 유형코드별 1개 기본 양식 자동 생성 후, -- 운영자가 과목·시스템별 양식을 추가 생성 INSERT INTO ncm_form_templates (form_id, name, type_code, target_systems, status) VALUES ('FORM-ANS-A', '눈높이 국어 뉴드림스용 입력 양식', 'ANS', '["ND"]', 'active'), ('FORM-ANS-B', '눈높이 수학 뉴드림스용 입력 양식', 'ANS', '["ND"]', 'active'), ('FORM-ANS-NH1', '눈높이 국어 해답지 입력 양식', 'ANS', '["NH"]', 'active'); -- ↑ type_code 'ANS'는 동일, 양식만 다르게 생성 -- 양식별 필드 구성(fieldDefs)도 과목 특성에 맞게 다를 수 있음

Phase 2 — 커리큘럼 자동 생성 (Day 3~7)

2

과목코드(SUBJECT_COURSE_DATA) → ncm_curriculum 자동 생성

기존 29개 과목코드 × 유형 조합에서 커리큘럼을 자동으로 생성합니다. 과목코드 1개 = 커리큘럼 1개로 1:1 초기 매핑 후, 이후 자유롭게 분리/병합 가능합니다.

과목코드 → 커리큘럼 자동 변환 로직
기존: M (수학) → courses:8, sets:80, types:[ANS,AN2,AN3,CBT,STDG]
      ↓ 자동 변환
신규: CUR-M-001 "수학 기초 과정"
  ├─ depths: [커리큘럼명(1), 과정(8), 세트(80), 콘텐츠(null)]
  ├─ depths[2].forms: [
  │   FORM-ANS-A, FORM-AN2-A, FORM-AN3-A,
  │   FORM-CBT-A, FORM-STDG-A
  │  ]
  └─ 자동 바인딩: boundCodes: ['M'], bindConfig: {M: {mode:'full'}}
-- Phase 2: 과목별 커리큘럼 자동 생성 (pseudo SQL) INSERT INTO ncm_curriculum (curriculum_id, name, depths, split_depth, status) SELECT 'CUR-' || s.code || '-001', s.course_name || ' 기초 과정', JSON_BUILD_ARRAY( JSON_BUILD_OBJECT('name','과목', 'count',1), JSON_BUILD_OBJECT('name','과정', 'count',s.courses), JSON_BUILD_OBJECT('name','세트', 'count',s.sets, 'forms', /* Phase 1에서 매핑된 양식 배열 */), JSON_BUILD_OBJECT('name','콘텐츠', 'count',null) ), 2, -- splitDepth: 세트 레벨 'active' FROM legacy_subjects s; -- 동시에 바인딩도 자동 생성 (기존 연결 유지) INSERT INTO ncm_content_bindings (curriculum_id, subject_code, target_system, bind_mode) SELECT 'CUR-'||code||'-001', code, 'ND', 'full' FROM legacy_subjects;

Phase 3 — 작업 데이터 이관 (Day 7~10)

3

기존 작업 → ncm_work_assignments 이관

기존 작업 데이터에 curriculum_id와 form_id를 역매핑하여 이관합니다. 기존 유형코드로 양식을 자동 조회합니다.

-- Phase 3: 기존 작업 → 신규 작업 이관 INSERT INTO ncm_work_assignments (curriculum_id, form_id, depth_type, assignee, range_from, range_to, status) SELECT 'CUR-' || w.subject_code || '-001', -- Phase 2에서 생성한 커리큘럼 f.form_id, -- Phase 1에서 매핑한 양식 w.type_code, -- 기존 유형코드 그대로 w.assignee, w.range_from, w.range_to, w.status FROM legacy_work_items w JOIN ncm_form_templates f ON f.type_code = w.type_code;

✅ 결론
바인딩 테이블은 사전에 생성되는 인덱스 테이블입니다. 과목코드로 커리큘럼을 찾는 것은 PK 기반 조회이므로 성능 차이는 무시할 수 있습니다.

오히려 하나의 커리큘럼을 여러 과목코드에 바인딩할 수 있으므로, 동일 콘텐츠를 복제하지 않고 재사용할 수 있어 스토리지 절감 + 유지보수 비용 감소 효과가 있습니다.
