1 핵심 한 장 요약
전체를 관통하는 핵심 메시지 하나.

콘텐츠는 과목코드를 몰라도 된다

현행: 유형코드(ANS, CBT...)가 콘텐츠의 키 → 과목코드와 유형코드가 강하게 결합
신규: 커리큘럼 + 입력양식이 콘텐츠의 키 → 과목코드는 나중에 "연결"만 하면 됨

비유: 상품(콘텐츠)과 판매채널(과목코드)
📦
콘텐츠 = 상품
상품은 상품대로 만들고
카탈로그(양식)에 따라 등록
⟷
🏪
과목코드 = 판매채널
어떤 매장에서 팔지는
나중에 "진열(바인딩)"하면 됨
같은 상품을 여러 매장에 진열 = 하나의 콘텐츠를 여러 과목코드에 바인딩
2 현행 vs 신규 구조 비교
유형코드 중심에서 과목코드 중심으로, 무엇이 달라지는가.

유형코드가 키 (현행)

과목코드 안에 유형코드가 종속되어 있음. 콘텐츠를 만들려면 먼저 "어떤 과목의 어떤 유형인지"를 알아야 함.

과목코드는 나중에 연결 (신규)

콘텐츠는 커리큘럼 + 양식으로 독립적으로 존재. 과목코드는 서비스 연결 시점에 바인딩.

콘텐츠 제작 영역 (①②③)

과목코드와 무관하게 독립적으로 진행

1
양식 관리자가 필드 풀에서 양식 구성
예) "해답지 양식" = 문항번호 + 정답 + 배점 + 해설
2
커리큘럼 설계자가 Depth 구조 설계 + 양식 연결
예) 과목 › 과정 › 세트 › 학습일 + "해답지 양식" 매핑
3
작업자가 커리큘럼 기준으로 콘텐츠 등록
예) 세트 1~40 담당, 양식에 맞춰 데이터 입력

서비스 연결 영역 (④⑤)

완성된 콘텐츠에 과목코드를 연결하여 서비스 투입

4
과목코드 바인딩 — 완성 커리큘럼에 코드를 연결
예) "수학 기초 과정" 커리큘럼 → M(수학) 코드 바인딩
5
배포 — 바인딩 완료된 콘텐츠만 배포 가능
바인딩 없으면 배포 대상에서 제외 (안전장치)
핵심 포인트
외부 시스템에서 과목코드 - 과정 - 세트 정보가 내려오면,
과목코드를 키로 바인딩만 하면 콘텐츠가 즉시 연결됩니다.
콘텐츠를 다시 만들 필요가 없습니다.
4 실전 시나리오
실제 업무에서 어떻게 달라지는지 구체적으로.
📘 시나리오 A — 수학 해답지를 뉴드림스 + 눈높이에 동시 제공
❌ 현행 방식
1
뉴드림스용 M-ANS 해답지 콘텐츠 제작
2
눈높이용 M-ANS 해답지 동일 콘텐츠 복사해서 별도 제작
3
수정 발생 시 양쪽 모두 수정 (작업 2배)
결과: 동일 콘텐츠 2벌 관리, 수정 포인트 2배
✅ 신규 방식
1
"수학 기초 과정" 커리큘럼에 해답지 양식으로 콘텐츠 1벌 제작
2
뉴드림스 과목코드 M 바인딩
3
눈높이 과목코드 M-NH 바인딩 (같은 콘텐츠에 추가 연결)
결과: 콘텐츠 1벌, 수정 시 1번만 수정, 양쪽 자동 반영
🤝 시나리오 B — 외부 제휴사에 콘텐츠 제공
1
이미 제작 완료된 "수학 기초 과정" 커리큘럼이 있음
2
제휴사가 자체 과목코드 EXT-M01을 전달
3
바인딩 화면에서 EXT-M01 → "수학 기초 과정" 연결 (클릭 한 번)
4
배포 → 제휴사 시스템에 콘텐츠 제공 완료
콘텐츠를 다시 만들 필요 없음. 기존 콘텐츠에 코드만 연결하면 끝.

5 데이터 구조 비교
개발자를 위한 데이터 레벨 비교.

현행 데이터 흐름

// 콘텐츠 조회: 과목 + 유형으로 직접 접근 SELECT * FROM content WHERE subject_code = 'M' AND type_code = 'ANS' AND course = '기초수학' AND set_no = 1; // 문제: 과목코드가 바뀌면? // → 콘텐츠 전체 마이그레이션 필요 // → 다른 시스템에 제공? 복사해야 함

신규 데이터 흐름

// 콘텐츠 조회: 커리큘럼 + 양식Key로 접근 SELECT * FROM ncm_content_data WHERE curriculum_id = 'CUR-001' AND form_key = 'ANS'; // 서비스 연결: 바인딩 테이블만 조회 SELECT b.subject_code, c.* FROM ncm_content_bindings b JOIN ncm_content_data c ON c.curriculum_id = b.curriculum_id WHERE b.subject_code = 'M';

6 4/13 회의 피드백 대응
개발팀에서 제기한 주요 우려사항과 이 구조에서의 해결 방안.

우려 1: 브릿지 테이블 성능 — "하루 수백만 조회 API에 조인이 항상 가야 해서 성능 저하"

✅ 해결

콘텐츠 조회 시에는 바인딩 테이블을 조인하지 않습니다.
콘텐츠는 [curriculum_id][form_key] Path로 직접 접근합니다.
바인딩 테이블은 "이 과목코드에 어떤 커리큘럼이 연결됐나?"를 확인하는 관리 목적에서만 사용됩니다.
실 서비스 API는 curriculum_id를 이미 알고 있으므로 바인딩 조인 불필요.

우려 2: 콘텐츠Data는 form_id를 알아야 할 이유가 없다

✅ 동의 + 대안

맞습니다. form_id 대신 양식 Key(ANS, CBT 등)를 선택적으로 부여합니다.
Key가 있으면: [curriculum_id][form_key]로 조회
Key가 없으면: [curriculum_id][content_id]로 조회
→ form_id가 콘텐츠에 직접 노출되지 않습니다. 4/13 회의의 "2번 대안"과 동일한 방향입니다.

우려 3: "v7 전환 시점에 모든 시스템은 전부 레거시, 언제 생길지 모를 V7 사용 대상을 위해 브릿지 전략은 보수적"

✅ 관점 전환

이 구조의 핵심은 "미래를 위한 투자"가 아니라 "현재의 관리 비용 절감"입니다.
현행: 과목 30개 × 유형 10개 = 300개 조합을 각각 관리
신규: 커리큘럼 N개 + 양식 M개 → 바인딩으로 300+ 코드 대응
콘텐츠 1벌로 여러 서비스를 커버하므로 운영 인력과 수정 비용이 즉시 줄어듭니다.

제안: 양식 상속 전략 — STD_ANS → M_ANS, SO_ANS 분기

✅ 수용 — 두 가지 옵션 모두 지원

옵션 A (상속): 표준 양식을 참조하되, 하위 양식에 커스텀 필드 추가. 표준 변경 시 하위에 영향.
옵션 B (복사): 표준 양식을 복사하여 독립 양식 생성. 변경 영향 없지만 공통 수정 시 개별 관리.
현재 프로토타입의 "필드 풀 + 양식 편집기"가 이 두 가지를 모두 지원하는 구조입니다.
ANS Key는 유니크 제약으로, 상속 외에는 동일 Key 부여 불가 (중복 방지).
