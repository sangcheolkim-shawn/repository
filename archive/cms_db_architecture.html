CMS DB 아키텍처 — 정형 + 비정형 하이브리드

필드를 자유 구성해도 DB에 문제가 없는 구조. 정형 데이터는 RDBMS 관계로, 유연 데이터는 JSONB로, 비정형은 별도 테이블로 관리합니다.
📊 ERD 구조
🔄 데이터 흐름 시뮬레이션
🔍 쿼리 테스트
⚠️
기존 방식의 문제: 유형마다 고정 테이블(content_an2, content_cbt...)을 만들면, 운영자가 필드를 추가할 때마다 ALTER TABLE이 필요하고, 과목별 오버라이드 시 같은 유형인데 컬럼 수가 달라집니다. 19개 유형 × 29개 과목 = 관리 불가능.
✅
해결: JSONB 하이브리드: 고정 관계(subjects→courses→sets→content_items)는 RDBMS로, 가변 필드 값(field_values)은 JSONB로 저장합니다. 비정형 컨텍스트(메모/태그/링크)는 별도 테이블로 분리합니다.

테이블 구조

정형 테이블 (RDBMS 관계)

📚 subjects

PK code VARCHAR(10)
* name VARCHAR(50)
FK parent_code → subjects

📖 courses

PK id SERIAL
* course_code VARCHAR(20)
FK subject_code → subjects
   source 'batch' | 'manual'

📁 sets

PK id SERIAL
* set_code VARCHAR(20)
FK course_id → courses

📋 content_items

PK id SERIAL
FK set_id → sets
* type_code 'AN2','FLC'...
   status idle → live
   assignee VARCHAR
유연 + 비정형 (JSONB & Text)

⭐ content_data JSONB

PK id SERIAL
FK content_item_id → content_items
* row_index INT
* field_values JSONB ← 핵심!
   file_path VARCHAR

💬 context_notes 비정형

FK content_item_id → content_items
* text TEXT (전문검색)
   is_pinned BOOLEAN

🏷️ context_tags 비정형

FK content_item_id → content_items
* tag VARCHAR(50)
메타 테이블 (필드 정의 관리)

🧩 field_pool → type_fieldsets → subject_overrides

글로벌 필드 정의 → 유형별 기본셋 → 과목별 커스텀
이 3개 테이블로 "어떤 필드를 보여줄지" 결정

JSONB가 해결하는 핵심 문제

❌기존: 유형별 고정 테이블
-- 유형마다 테이블이 필요 CREATE TABLE content_an2 (no, answer, score, note); CREATE TABLE content_cbt (section, content, keyword); CREATE TABLE content_flc (front, back, image); -- 19개 테이블... -- 영어 AN2에 '언어' 추가하면? ALTER TABLE content_an2 ADD COLUMN lang VARCHAR; -- 수학에선 안 쓰는데 컬럼이 생김 😱
✅개선: JSONB 단일 테이블
-- 테이블 1개로 모든 유형 처리 CREATE TABLE content_data ( content_item_id INT, row_index INT, field_values JSONB ← 핵심! ); -- 영어 AN2 {"no":"1", "answer":"③", "score":"5", "lang":"EN"} -- 수학 AN2 (비고 없이, 언어 없이) {"no":"1", "answer":"③", "score":"5"} -- ALTER TABLE 불필요! 😊
